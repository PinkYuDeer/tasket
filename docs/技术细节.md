# 技术细节

## 总览

此仓库为 `Tasket`（中文名：任物篮，modId：`tasket`），一句话介绍为“把任务装进任物篮，享受一步步填满小篮子的治愈感。” / “Tasket – a cozy task basket for your Minecraft journey.” 今骨架已具五端：Forge 生命周期入口、SQLite 持久层、任务领域模型与 DAO、ModularUI2 客户端界面、FML 网络通道。其现状宜判为“单机可跑、多人未定型”：数据模型与 UI 已先行，服务端权威、同步协议、权限与事务尚须补齐。

## 技术栈

- 构建：GTNH Convention Plugin，RetroFuturaGradle，`settings.gradle.kts` 载入 `com.gtnewhorizons.gtnhsettingsconvention`。
- 运行目标：Minecraft `1.7.10`，Forge `10.13.4.1614`。
- Java：`enableModernJavaSyntax = jabel`，可用较新语法，仍降至 Java 8 字节码语义。
- 主要依赖：`sqlite-jdbc`、`ModularUI2`、`Lombok`、`Reflections`、`Kryo`、`Gson`、`jsr305`。
- CI：GitHub Actions 已接 GTNH build/test 与 tag release。

## 目录与分层

- `com.pinkyudeer.tasket.Tasket`：`@Mod` 入口，所有生命周期转交 `CommonProxy` / `ClientProxy`。
- `core/`：配置、事件、网络注册、客户端/服务端代理。
- `config/` 与 `helper/`：配置封装、文件路径、玩家、命令与通用反射/类型转换工具。
- `db/`：自研注解式 SQLite 层，含 `@Table`、`@Column`、`@Reference`、`@FieldCheck`、SQL Builder、`EntityHandler` 与 `SQLiteManager`。
- `task/`：待办事项领域核心。`entity/` 放 `Task`、`Player`、`Team`、`Tag`；`entity/record/` 放记录表；`dao/` 放各表 DAO；`service/` 放用例层。
- `gui/` 与 `render/`：ModularUI2 界面、面板、控件、Shader 与模糊背景。
- `network/`：类 BetterQuesting 的 NBT 包协议、分片重组、包族分派与发送工具；旧 `NetWorkData`/`NetWorkHelper` 暂为兼容残件。
- `client/`：客户端只读同步缓存与 client-only 网络 handler。
- `src/main/resources/assets/tasket/`：语言、贴图、Shader、物品/方块资源。

## 启动与运行流

1. `Tasket` 接 Forge 生命周期事件，委托给代理。
2. `preInit` 中加载配置、初始化模组目录、注册创造标签、方块、物品；客户端另注册快捷键与主题。
3. `init` 中注册配方、世界事件、玩家登录事件、SimpleNetworkWrapper 服务端消息；客户端另注册收包 handler 并初始化模糊渲染。
4. `serverStarting` 启动 `ServerTaskScheduler` 并注册 `/task` 命令。
5. 主世界 `WorldEvent.Load` 时更新世界目录并初始化 SQLite。新世界则扫描 `task.entity` 下有 `@Table` 之类并建表；旧世界则将 `main.db` restore 至内存库，再执行轻量迁移。
6. 主世界保存或卸载时，将内存 SQLite backup 至世界目录下 `tasket/main.db`，卸载后关闭连接。
7. 玩家登录时调用 `PlayerDao.updateOrInsert` 写入或更新玩家资料，尝试刷新 BQ-linked 团队，并发送 `main_sync` reset；客户端随后请求全量同步。

## 数据模型

核心实体已较丰满：

- `Task`：含标题、描述、版本、可见性、任务类型、父任务、创建者、团队、点赞/关注/评论计数、优先级、重要度、紧急度、状态、创建/开始/结束/更新/截止/提醒/重复时间等。
- `Player`：含 UUID、玩家名、显示名、积分、等级、团队数、通知数、关注/点赞/评论统计、完成任务统计、角色、账号状态、登录与更新时间。
- `Team`：含团队名称、简介、加入码、所有者、成员/任务/标签统计、默认任务可见性、积分、等级、权限开关、请求计数；并新增 `syncSource`、`externalPartyId`、`externalTeamKey`、`syncStatus`、`lastSyncTime` 以承载本地/BQ/GTNHLib 外部成员源。
- `Tag`：用于任务、团队、玩家等对象归类。
- 记录表：`BaseRecord` 统一记录 id、创建时间、操作者；`TaskInteraction`、`PlayerInteraction`、`TeamMember`、`TeamRequest`、`TagLink`、`Notification`、`StatusChangeRecord` 承载协作、审计、通知、成员关系与标签关联。

须留意一处类型不一：`Task.id` 与 `Task.parentTaskId` 为 `String`，多数字段则用 `UUID`；而 `TaskInteraction.parentTaskId` 为 `UUID`。多人同步与查询 API 定型前，应统一 ID 表示，或明文规定边界转换。

## 持久层

此项目非用 JPA，乃自研轻量 ORM：

- `@Table` 指表名。
- `@Column` 指列名、默认值、主键、唯一、索引。
- `@Reference` 指外键实体与字段。
- `@FieldCheck` 生成 SQLite `CHECK` 约束，如 UUID 长度、枚举值、范围、字符串长度。
- `SQLUtils` 由字段类型生成 SQLite 类型，枚举与 UUID 以 `TEXT` 入库，`LocalDateTime` 为 `TIMESTAMP`，`Duration` 为 `REAL`。
- `CreateTableBuilder` 可按依赖拓扑排序建表；`InsertBuilder`、`SelectBuilder`、`UpdateBuilder`、`DeleteBuilder` 负责 CRUD；`AlterTableBuilder` 已备复杂表结构变更骨架。
- `EntityHandler` 以反射把 `ResultSet` 映射回实体，借 `UtilHelper.convertValue` 做类型转换。

此层优点是无重依赖、贴合 1.7.10 环境；短板是事务、连接/语句生命周期、迁移版本、并发访问、异常传播、测试覆盖尚弱。`SQLiteManager.executeSafeSQL` 查询时直接返回 `ResultSet`，调用方无法清晰持有并关闭 `PreparedStatement`；后续宜抽象查询回调或结果收集器。

## 业务层

`TaskService` 已承接创建、更新、状态变更、完成、关闭、删除、子任务查询等用例，并新增 `PermissionContext` 作为团队任务入口。`TeamService` 现为本地多团队、BQ-linked 团队与 GTNHLib-linked 团队之服务层入口，负责创建、邀请、申请、接受、踢出、退出、转让队长、关联/解除外部团队与成员同步。`TagService` 已承接任务标签创建、绑定、解绑、按任务查询、按标签查任务与可见性校验，任务同步包会附带标签 NBT，供 GUI 后续接入。核心只依赖 `TeamProvider`，BQ 与 GTNHLib 均以反射 provider 隔离；BQ 保留 int party，GTNHLib 使用 `teamName` 作为 `externalTeamKey`。`StatusChangeRecord` 可记录状态流转，方向正确。然当前仍有数项需固化：

- 权限尚未全量收口：团队创建、成员管理与团队任务创建已进入上下文，但旧 GUI、命令与部分任务编辑路径仍需改为服务端 action。
- 事务已入核心写链路：团队成员流转、状态变更与删除已走事务；后续仍须覆盖评论、标签、奖励、通知等派生写入。
- 版本控制尚浅：`Task.version` 会递增，但多人并发冲突尚未形成乐观锁契约。
- `TaskService` 已用查询后深拷贝处理状态变更，后续编辑路径也应沿用此模式，避免差异更新语义偏离真实旧值。
- 计数与派生字段尚未闭环：点赞、关注、评论、完成数、团队统计、邀请/申请计数等字段有设计，更新链路尚缺完整事务。

## 客户端界面

GUI 骨架已可用：

- `KeyBindGuiHandler` 监听快捷键，打开任务界面。
- `TaskScreen` 包装 ModularUI2 界面并做关闭淡出。
- `MainPanel` 展示任务列表、子任务树、展开/折叠、排序、显示/隐藏完成项、新建入口，读源已改为 `TaskClientStore`。
- `TaskFormPanel` 负责新建任务或子任务，选择重要度/紧急度，写入已改为发送 `task_action`。
- `TaskDetailPanel` 负责查看、编辑、变更状态、完成、删除、管理子任务，读子任务走客户端缓存，写操作走 `task_action`。
- `ShaderDrawable`、`BlurHandler`、`GLShaderDrawHelper` 提供圆角面板、阴影、模糊背景与自定义 Shader。

当前 GUI 主路径已不直接调用 `TaskService`；客户端发请求，服务端验证并写库，服务端再回推同步。后续仍须补“待确认操作/失败回滚/自动刷新”细节，避免请求尚未回包时界面短暂显示旧数据。界面文本亦多为英文硬编码，后续须迁至 `lang`。

## 网络层

网络层已改为类 BetterQuesting 架构：

- `PacketHandler` 建立 `tasket` SimpleNetworkWrapper 通道，只注册单一 `TasketPacket`。
- `TasketPacket` 承载 `NBTTagCompound`，包内 `ID` 为 `ResourceLocation`，由 `PacketTypeRegistry` 分派至具体 handler。
- 客户端收包 handler 已移至 `client.network.ClientTasketPacketHandler`，只由 `ClientProxy` 注册；common 侧不再直接引用 `Minecraft`。
- `PacketAssembly` 负责 NBT 压缩、`20480` 字节切片与按玩家 UUID 重组，故大同步包不再依赖 Kryo `Object`。
- `PacketSender` 提供 `sendToPlayers`、`sendToAll`、`sendToServer`、`sendToAround`、`sendToDimension`。
- `ServerTaskScheduler` 将服务端业务 handler 切回 server tick 执行，避免网络线程直接写库。
- 固定包族为 `main_sync`、`task_action`、`task_sync`、`team_action`、`team_sync`、`invite_sync`、`error`。
- `task_sync` 已按 OP、创建者、公开任务与团队成员关系过滤可见任务。
- 客户端以 `TaskClientStore` 收同步数据，后续 GUI 应只读该缓存并通过 action 包发操作意图。

旧 `NetWorkData`/`NetWorkHelper` 仍留作兼容残件，后续可在 GUI 完成切换后删除。

## GTNHLib Team 学习结论

GTNewHorizons/GTNHLib#297 新增集中 team 系统，意图以后承接各 Mod 各自的团队模型。其要点为：

- `TeamManager` 以静态入口管理 team，可按玩家 UUID 取队伍，也可按 `teamName` 查队伍；玩家会被放入个人 team，合并时把被消费 team 的数据合并到存续 team。
- `Team` 以 `teamName` 为外部键，维护 owners、officers、members 三层集合；owner 与 officer 均可映射为 tasket `ADMIN`，普通 member 映射为 `MEMBER`。
- `ITeamData` 提供 `writeToNBT/readFromNBT/mergeData/markDirty`；各 Mod 经 `TeamDataRegistry.register(key, Supplier)` 给每个 GTNHLib team 挂自己的数据。
- tasket 已预留 `integration.gtnhlib`：`GtnhLibTeamProvider` 用反射读 `TeamManager`/`Team`，`GtnhLibTeamDataBridge` 注册 `tasket` data key。PR 未合并或旧版 GTNHLib 无此 API 时，provider 不可用且不会崩服。
- 因 GTNHLib 当前设计偏“玩家一队 + solo team”，而 tasket 本地模型仍为“玩家可属多个团队”，故本轮只把 GTNHLib team 当外部成员源；任务数据仍归 tasket SQLite，未来如要迁入 `ITeamData`，应写迁移器而非硬切存储。

## BetterQuesting 学习结论

BetterQuesting 之网络与组队有数点可借：

- 单一底层包：`PacketQuesting` 只传 NBT，真实业务由 `ID` 分派，新增协议不必新增 Forge discriminator。
- 大包切片：先压缩 NBT，再分片传输，接收端重组后才交给 handler。
- 主线程调度：服务端包 handler 不直接改数据库，而是排入 server tick。
- 登录全量同步：玩家登录先 reset 客户端缓存，再 bulk sync 设置、任务、队伍、邀请与名称。
- 队伍权限：party 成员为 `OWNER/ADMIN/MEMBER`，邀请独立保存，队伍删除时清理邀请。

tasket 采用其网络形态，但不照搬单队伍限制。本项目本地模型仍为“玩家可属多个团队”；关联 BQ 后，只把 BQ party 当作某个团队的外部成员源，且为单向同步。若未来 BQ 迁至 GTNHLib Team，核心服务不应改写，只需替换或收敛 provider。

## 已成之处

- Forge 生命周期、资源注册、命令入口、事件入口已经清晰。
- 领域模型覆盖较广，已预留任务、团队、标签、通知、互动、审计与积分统计。
- SQLite 内存库加世界文件持久化之思路适合频繁读写，亦便于世界级隔离。
- 自研 SQL Builder 与注解体系已能支撑初期 CRUD 与建表。
- GUI 已非空壳，已有列表、详情、表单、子任务与排序交互。
- `TeamServiceTest` 与 `TagServiceTest` 已用内存 SQLite 覆盖本地团队流转、外部团队只读、任务标签绑定/解绑/去重与团队标签权限。

## 主要风险

- 多人权威模型未定：数据写入须以服务端为准，客户端 GUI 只发意图与渲染结果。
- 外部组队集成必须隔离：核心服务不得直接 import BQ 或 GTNHLib Team 类，缺席时需退化为纯本地多团队。
- Dedicated Server 仍须实机验证：本轮已将 `EventHandler`、`ModFileHelper` 与 `TasketPacket` 的客户端路径拆至 proxy/client handler，但发布前仍需跑无 BQ 专服启动测试。
- 文档有旧债：`docs/TaskSqlHelper使用指南.md` 所述 `TaskSqlHelper.entity(...)`、`ConvenienceMethods` 与现代码不符，宜更新或归档。
- 测试仍不足：已有团队与任务标签服务测试，但 DB Builder、类型转换、迁移、TaskService 全流程仍需补自动验证。
- 数据迁移仍初阶：已引入 `schema_version` 与幂等添加列迁移，但无复杂表重建、回滚策略与自动测试。
- 资源释放不清：查询返回 `ResultSet`，`PreparedStatement` 生命周期不明；长期运行服或高频查询下需改。
- 构建配置已清理：`usesMixins=false` 且无实际 mixin 类，遗留 `mixins.tasket.json` 已删除。
- DAO 重复度高：各 DAO 大量同形方法，可保留门面，但底层宜抽出泛型基类或模板，以免扩表时维护成本增高。
- 指令入口尚多 TODO：`/task add/remove/list/update` 未接服务层。

## 建议定案

短期勿先扩 UI 花活，宜先定“服务端权威 + 稳定 DB + 明确协议 + 多团队权限”四事。技术主线如下：先补构建与 DB 测试，再固化 `TaskService`/`TeamService` 契约，然后让 GUI 全部走网络状态。如此后续团队、通知、BetterQuesting、奖励特效皆可附着其上，而不致反复重构。
